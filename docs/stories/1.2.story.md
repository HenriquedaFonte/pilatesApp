# Story 1.2 — Edge Function: Soft Delete + Audit Log em `manage-student`

**Epic:** 1 — Proteção de Dados (Fase 1 — MUST)
**Status:** Done
**Estimativa:** M (3–5h)
**Tipo:** API / Backend (Supabase Edge Function)
**Depende de:** Story 1.1 (tabelas `audit_log` e colunas `deleted_at`/`deleted_by` devem existir)

---

## User Story

**Como** professora do estúdio,
**Quero** que ao excluir um aluno, ele seja movido para um estado de "excluído" (soft delete) com registro de quem executou a ação,
**Para que** eu possa recuperar os dados e rastrear a responsabilidade caso algo ocorra por engano.

---

## Contexto para o Dev

A edge function `supabase/functions/manage-student/index.ts` atualmente executa hard delete cascata (apaga tudo). Esta story modifica o fluxo de `action: 'delete'` para:
1. Capturar snapshot do perfil antes de qualquer modificação
2. Inserir registro no `audit_log`
3. Executar soft delete em `profiles` (SET deleted_at, deleted_by) em vez de DELETE
4. **Não apagar** `check_ins`, `balance_history` — eles ficam como histórico

Também adiciona um novo `action: 'restore'` para reverter o soft delete.

---

## Acceptance Criteria

- [ ] **AC-1:** `action: 'delete'` não apaga o registro de `profiles` — faz `UPDATE SET deleted_at = NOW(), deleted_by = actor_id`
- [ ] **AC-2:** Antes do soft delete, insere registro em `audit_log` com: `actor_id`, `actor_email`, `action = 'student.soft_delete'`, `target_id`, `target_name`, `target_email`, `snapshot_data` (JSON do perfil completo)
- [ ] **AC-3:** `check_ins` e `balance_history` **NÃO são deletados** no fluxo de soft delete
- [ ] **AC-4:** `student_class_link` é deletado (aluno não deve aparecer em turmas após exclusão)
- [ ] **AC-5:** `email_notifications` pode ser mantido ou deletado — decisão: manter para histórico
- [ ] **AC-6:** Novo `action: 'restore'` reverte o soft delete (`UPDATE SET deleted_at = NULL, deleted_by = NULL`) e registra no `audit_log` com `action = 'student.restore'`
- [ ] **AC-7:** `action: 'hard-delete'` (novo) executa o cascade delete original com log obrigatório no `audit_log` antes de executar
- [ ] **AC-8:** Erro retorna JSON `{ error: string }` com status 400 (sem mudança no padrão existente)
- [ ] **AC-9:** Auth user (`auth.admin.deleteUser`) **NÃO é chamado** no soft delete — apenas no hard delete

## Riscos

| Risco | Probabilidade | Impacto | Mitigação |
|-------|---------------|---------|-----------|
| Audit log falha silenciosamente antes do soft delete | Baixa | Alto | INSERT em audit_log lança exceção — qualquer falha aborta a operação inteira (AC-8) |
| Service role bypass não registra actor real | Alta (histórico) | Médio | Actor sentinel `'service_role'` documentado e logado — rastreável no audit_log |
| `action: 'delete'` chamado por código antigo que espera hard delete | Média | Médio | Comportamento muda para soft delete — Story 1.3 atualiza UI em paralelo; sem chamadas externas conhecidas |
| check_ins ou balance_history deletados acidentalmente | Baixa | Alto | AC-3 e AC-5 explícitos; CodeRabbit Focus Area verifica ausência de `.delete()` nessas tabelas |
| Restore de aluno com email já recriado por outro cadastro | Baixa | Alto | Fora do escopo desta story — tratado como edge case na Story 2.1 |

## Dev Notes

### Arquivo a Modificar
- `supabase/functions/manage-student/index.ts`

### Fluxo Atual do `action: 'delete'`
```
1. Delete student_class_link WHERE student_id
2. Delete check_ins WHERE student_id
3. Delete balance_history WHERE student_id
4. Delete email_notifications WHERE student_id
5. Delete profiles WHERE id
6. auth.admin.deleteUser(studentId)
```

### Novo Fluxo do `action: 'delete'` (soft delete)
```
1. SELECT * FROM profiles WHERE id = studentId → capturar snapshot
2. INSERT INTO audit_log (actor_id, actor_email, action, target_id, target_name, target_email, snapshot_data)
3. DELETE FROM student_class_link WHERE student_id (remover de turmas)
4. UPDATE profiles SET deleted_at = NOW(), deleted_by = actorId WHERE id = studentId
   (NÃO apagar check_ins, balance_history, email_notifications)
```

### Novo `action: 'restore'`
```
1. SELECT * FROM audit_log WHERE target_id = studentId ORDER BY created_at DESC
   (para verificar que o aluno estava soft-deleted)
2. UPDATE profiles SET deleted_at = NULL, deleted_by = NULL WHERE id = studentId
3. INSERT INTO audit_log (action = 'student.restore', ...)
```

### Novo `action: 'hard-delete'`
```
1. SELECT * FROM profiles WHERE id = studentId → capturar snapshot final
2. INSERT INTO audit_log (action = 'student.hard_delete', snapshot_data)
3. Executar o cascade delete original (student_class_link, check_ins, balance_history, email_notifications, profiles)
4. auth.admin.deleteUser(studentId)
```

### Captura do Actor
O actor já é obtido no fluxo existente via:
```typescript
const { data: { user } } = await userClient.auth.getUser()
// user.id = actorId
// user.email = actorEmail
```

### Inserção no audit_log
```typescript
const { error: auditErr } = await supabaseClient
  .from('audit_log')
  .insert({
    actor_id: user.id,
    actor_email: user.email,
    action: 'student.soft_delete',
    target_id: studentId,
    target_name: profileSnapshot.full_name,
    target_email: profileSnapshot.email,
    snapshot_data: profileSnapshot
  })

if (auditErr) {
  throw new Error(`Failed to write audit log: ${auditErr.message}`)
}
```

### Nota sobre Service Role bypass
Quando `isServiceRole = true`, o `user` não está disponível (bypass de diagnóstico). Neste caso, `actor_id` deve ser um UUID sentinel (`'00000000-0000-0000-0000-000000000000'`) e `actor_email` = `'service_role'`.
